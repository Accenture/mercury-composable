### Finding — the ElasticQueue two-tier throughput cliff (laptop, file store, 200k ops)
| scenario | throughput | mean | note |
|---|--|--|--|
| RPC 1→50 | 21k | 0.047 ms | baseline, in-flight=1 |
| RPC 50→50 | **165k** | 0.30 ms | **in-memory tier** (queue ≤20) |
| Callback 50→50 paced | 39k | **0.47 ms** | healthy async, no spill |
| RPC 100→50 | 75k | 1.33 ms | **disk-spill tier** (queue ~50 >20) |
| Callback flood→50 | 78k | 25 ms | disk-spill tier |

**Crossing the 20-event memory buffer roughly halves throughput:** in-memory ≈165k ops/s vs disk-spill
≈75–78k ops/s. Both spill scenarios (RPC 100→50 and the callback flood) land at the same ~75–78k ceiling —
strong evidence it's the spill path, not the workload. **Correction to the going-in assumption:** RPC 100→50
(2:1 oversubscription) does *not* stay out of back-pressure — ~50 requests queue (>20), so it spills; latency
rises ~4.6× (not the naive 2×) and throughput drops to the spill ceiling. Eric's call: **categorize it as
back-pressure** (it demonstrates back-pressure via oversubscription, complementing the flood). Paced async is
sub-ms (0.47) vs the flood's 25 ms — ~53× — showing healthy vs overloaded on the same 50 consumers. **0
failures / 0 loss everywhere** — back-pressure degrades predictably (Little's law), never collapses.
**Takeaway:** well-architected = keep in-flight ≤ consumers to stay in the fast in-memory tier; scale past a
single route's ceiling horizontally (more routes), per the earlier per-route-dispatch finding.

### Store A/B + the mixed-workload isolation win (the production recommendation)
Ran the suite file-vs-bdb (`-Delastic.queue.store=bdb`). **Honest surprise: on a single isolated route, BDB
(in its native loop mode) is competitive-to-faster** — higher throughput/lower median in several scenarios
(e.g. RPC 100→50: bdb 122k vs file 75k), because inline dispatch avoids the vthread mailbox hand-off and
BDB's log-structured spill batches well. File's single-route edge is **tail latency** (RPC 100→50 max: file
3.5ms vs bdb 13.7ms). So the benchmark does NOT validate "file is faster" — BDB is a genuinely fast KV store
(Eric's read: "we picked a very fast KV store despite some over-engineering").

**But a single isolated route is blind to the actual migration driver.** Added a **mixed-workload** scenario
(`1bdf64a8`): a latency probe (paced RPC on a SEPARATE route `benchmark.probe`) measured WHILE a background
flood hammers `benchmark.worker`. **A/B probe latency under the flood:**
- **file+vthread:** mean 0.16ms, p99.9 1.54ms, **max 1.6ms** — spill off-loop, probe unaffected.
- **bdb+loop:** mean 1.32ms (**8.4×**), p99.9 12.1ms, **max 24.6ms (15×)** — spill inline on the shared event
  loop, leaking into the unrelated probe.

**This is the decisive, measured production argument.** Full picture: isolated throughput/median → BDB
competitive-to-faster; tail under spill → file better; **latency isolation under concurrent load → file
decisively better (8–15×).** Real production is multi-route with shared-loop contention, so **recommend
file+vthread as the production default** (isolation/robustness + portability + VT-safety), which the flipped
default / PR #137 deliver; BDB stays a one-flag fallback. Reports: `benchmark-report-{file,bdb}.html`.
