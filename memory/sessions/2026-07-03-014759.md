### The load-bearing design decision (non-obvious)

The flow's `FlowInstance.cid` is **also the HTTP response-routing key** ŌĆö REST automation awaits a reply
whose correlationId equals the internal per-request UUID (`requestId`), matched via the `contexts` map. My
first cut set the flow cid = the business X-Correlation-Id, which broke reply matching ŌåÆ **every HTTP flow
timed out** (65/70 FlowTests failed). Fix: **decouple** ŌĆö added `FlowInstance.correlationId` (business,
ŌåÆ `model.cid` + `my_correlation_id`) distinct from `FlowInstance.cid` (routing/reply, unchanged). The
business cid rides a new launch-event header `correlation_id` (constant `EventScriptManager.CORRELATION_ID`),
set by `FlowExecutor` (new 6-arg `launch` overload; defaults businessCid = routing cid for non-HTTP callers)
and read by `EventScriptManager`. `HttpToFlow` passes `event.getCorrelationId()` (requestId) as routing cid
and X-Correlation-Id as businessCid. Subflows inherit the parent's business cid via the same header. For
Kafka the two coincide (no requestId indirection), so no Kafka behavior change.

### Feature-complete extension (Eric: "proceed to make it feature complete")
