## Diagnosis — not a framework bug

1. **The behavior change is real, deliberate, and documented**: v4.5.0's breaking
   tracing/correlation cleanup (CHANGELOG) removed `trace.http.header` /
   `trace.http.legacy.header.enabled` — before 4.5.0, `X-Correlation-Id` could double
   as the trace id, which is almost certainly what made 4.4.11 "work" for them
   (gateways/apps commonly forward X-Correlation-Id). Since 4.5.0 the trace id travels
   ONLY as `X-Trace-Id` / W3C `traceparent`, and is never echoed back.
2. **The 4.6.3 propagation code is correct** (verified against the v4.6.3 tag —
   AsyncHttpClient outbound stamping and HttpRouter inbound continuation are the same
   logic as main). Their traceparent value is even our javadoc example and parses fine.
3. **Likely field misconfiguration**, two candidates to check:
   (a) `tracing: true` missing on a rest.yaml endpoint entry (per-endpoint, default
   false — without it the endpoint ignores inbound trace headers entirely);
   (b) the app-to-app call not going through `async.http.request` — automatic header
   stamping only happens there; a Spring RestTemplate/WebClient call must forward
   `traceparent` itself.
