## Conflated-header id divergence — field edge case fixed

Eric validated the legacy conflation config (`http.trace.id.header=X-Correlation-Id`)
end-to-end with a cross-app test (composable-example → lambda-example): supplied header
works perfectly (one id in both apps' telemetry). Edge case he caught: WITHOUT the
header, the wire carried `traceparent` with one generated id and `x-correlation-id`
with ANOTHER — the divergence I predicted in the 2026-07-13 analysis, now visible.

**Root cause:** two independent generators at the HTTP ingress — HttpRouter resolves the
cid on the PARSED request (generates UUID-A when absent) while getTraceId reads the RAW
request (generated cid invisible → UUID-B). Same shape in KafkaFlowConsumer.

**Fix (`fix/conflated-header-id-unification`, commit `a9742e63`):** when the RESOLVED
trace-id and correlation-id header names collide (case-insensitive, per-entry or global)
and the shared header is absent, the edge yields ONE id — the trace id is authoritative
(honors inbound traceparent) and the cid adopts it. Applied to both HttpRouter and
KafkaFlowConsumer. Supplied headers untouched (supplied feeds both; supplied+traceparent
honors each channel). Tests: /api/conflated/probe fixture + 3 contracts (generated=one
id; supplied=both; traceparent-adopted=both). Observability guide now documents legacy
conflation as SUPPORTED with the one-id guarantee. platform-core + event-script +
minimalist-kafka suites green.

**Supersedes part of the 2026-07-13 conflation answer:** "when absent, the two ids
diverge by design" no longer holds for CONFLATED names (one id guaranteed); it still
holds for DISTINCT names (two concepts, by design).

Field context: devops runs the conflation config short-term for backward compatibility
while infra allow-lists traceparent + X-Trace-Id (+ X-Correlation-Id already) at the API
gateway and K8s Istio.
