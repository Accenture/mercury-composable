### Applied (6)

- **mk#1 — producer failure visibility.** `KafkaRequestPublisher.publish` now passes a send-completion
  callback that logs async delivery failures. Stays drop-n-forget (no blocking); just ends the silent-masking.
- **mk#2 — consumer flow-failure = bounded retry then dead-letter.** New `RetryPolicy` record
  (`maxRetries`, `backoffMs`, `deadLetterPublisher`). `KafkaFlowConsumer.routeToFlow` retries on
  `ExecutionException`, then publishes the original message to `<topic>.dlq` (headers preserved +
  `dlq.origin.topic` / `dlq.error`) and commits — so a poison message is neither lost (the old behaviour:
  log + commit) nor reprocessed forever. Flow call extracted to a package-private `invokeFlow` seam for
  engine-free unit testing. Config: `kafka.flow.max.retries` (default 3), `kafka.flow.retry.backoff.ms`
  (default 500); wired through `KafkaFlowAutoStart` → `KafkaFlowAdapter` → consumer, reusing the shared
  publisher as the DLQ sink.
- **mk#3 — fail-fast config validation.** `KafkaFlowAdapter` validates the whole `consumer` list before
  opening any consumer; a non-map entry / missing-or-blank `topic` / missing-or-blank `flow` now throws
  `IllegalArgumentException` (was: silently skipped).
- **soa#1 — TOCTTOU capacity race.** `PendingRequests` reserves the slot with an `AtomicInteger`
  (increment-then-check, decrement on reject/remove) so concurrent `register` can't oversubscribe `maxPending`.
- **soa#2 — explicit Redis cleanup on success.** `ReturnRouteStore.cleanup` DELs both `request:`/`response:`
  keys; `ReturnRouteCoordinator.awaitResponse` calls it on the success path (and after final-read recovery)
  instead of waiting out the TTL.
- **soa#3 — lifecycle guards.** `start()` rejects double invocation; `close()` `shutdown()`s `signalWorkers`
  and awaits termination (5s) before `shutdownNow()`, so in-flight signal handlers finish before the
  connection is forced shut.
