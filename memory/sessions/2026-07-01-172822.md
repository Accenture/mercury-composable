### Part 1 — event-script-engine (the core ask)

- **New `SimpleTypeMatchingConverter`** (`com.accenture.automation`): converts every documented
  `model.key:type` form to its `f:type(...)` simple-plugin equivalent — the simple no-arg types
  (text/binary/int/long/float/double/boolean/uuid/length/b64), negate (`!`), `and`/`or`, `substring`,
  `concat`, and the `boolean(value[=true|false])` value-matching variant (→ `eq`/`ne`, or `isNull`/`notNull`
  for the documented `null` keyword test). Unrecognized/malformed qualifiers are left unconverted so the
  still-intact legacy resolution in `DataMappingHelper` keeps handling them (deliberate safety net — see
  the `model.text:unknown` and unbalanced-paren test cases).
- **`CompileFlows`** now runs every normalized input/output mapping line through the converter before
  validating/storing it, so `Task.input`/`Task.output` never contain the old syntax by the time
  `TaskExecutor` sees them.
- **`TaskExecutor`**: this is the actual deprecation — removed `getModelTypeIndex`, the colon branches in
  `setRhsElement`/`removeModelElement`, and `getValueFromNonExistModel`/`getNullBooleanValue` (traced these
  out carefully and confirmed they were **dead code** — every reachable case was already handled earlier in
  `DataMappingHelper.getValueByType` before falling through to them). TaskExecutor no longer parses
  colon-type syntax at runtime at all.
- **New `ne` plugin** (`NotEqualsOperator`, mirrors `EqualsOperator`) — needed to complete the 1:1 mapping
  for `boolean(value=false)`; Eric confirmed via AskUserQuestion (recommended option) rather than the
  temp-model-variable-expansion or leave-unconverted alternatives.
- **Real bug fix surfaced along the way:** `TypeConversionUtils.convertInteger/Long/Float/Double` used to
  `Integer.parseInt`-style throw on non-numeric input; the deprecated `:int` etc. semantics is
  "**-1 if not numeric**" (documented). Fixed to delegate to `Utility.str2int` etc. (graceful fallback) —
  this is what an existing test caught (`greetings.yml` exception-path flow uses a non-numeric path param
  through `model.user:int`, which used to gracefully return -1 and now would have thrown via the plugin
  without this fix).
- **Also fixed:** `DataMappingHelper.getValueFromSimplePlugin`'s plugin-argument comma-split was naive
  (didn't respect nested parens) — a `text(,)` argument inside `f:concat(...)` would have been mis-split.
  Added a paren-depth-aware `splitTopLevelArguments`. Pre-existing latent bug, unrelated to the deprecation
  itself but required for the `concat` conversion path to actually work (greetings.yml already exercises
  this exact case).
- **4 test-fixture edge cases removed** (`type-matching.yml` + `FlowTests.java`): `and(nothing)`,
  out-of-bounds/malformed-arg `substring`, and `substring` on binary data. These specifically probed the
  *old* mechanism's soft-failure/graceful-degradation quirks on malformed input — the new plugin path
  validates arguments strictly instead, so the premise of those tests no longer holds. Not a coverage
  regression; they tested the deprecated mechanism's own error-recovery behavior.
- **Docs:** `syntax.md` marks "Simple type matching" deprecated with a note on auto-conversion; added
  missing `ne`/`notNull`/`not` rows to the built-in plugins table (pre-existing doc gap, fixed in passing).
- New unit test `SimpleTypeMatchingConverterTest` (9 cases) + `ne` added to `SimplePluginLoaderTest`'s
  sub-millisecond benchmark. **event-script-engine: 115 tests green.**

### Part 2 — minigraph-playground-engine (impact analysis → new feature, both requested by Eric)

Two research passes confirmed minigraph reuses the *exact same* `DataMappingHelper` mini-DSL (including the
colon syntax — a real test fixture, `hellojs.json`, already contains `model.number:int -> hello.xyz`), but
had **zero** compile-time gate: `GraphExecutor.getGraphModel()` re-parsed the graph JSON from disk on
*every request*, and every data-mapping string was re-interpreted on *every node execution*, with no
upfront validation.

- **New `CompileGraph`** (`@BeforeApplication`, minigraph-playground-engine, which already depends on
  event-script-engine): opt-in via a `graph.model.automation` manifest (mirrors `yaml.flow.automation`).
  For each listed graph: converts deprecated colon syntax in `mapping`/`input`/`output`/`for_each` node
  properties via the same `SimpleTypeMatchingConverter`, sanity-checks `->` presence, and structurally
  validates via `MiniGraph.importGraph()` (reusing minigraph's own canonical validator rather than
  reimplementing skill-specific namespace rules — deliberately did **not** reuse event-script's
  `validInput`/`validOutput` here after confirming minigraph's per-skill namespace rules, e.g. fetcher
  input/output/dictionary, genuinely diverge and would false-positive-reject valid configs). Invalid
  graphs are logged + skipped, not fatal.
- **New `CompiledGraphs`** registry (mirrors `Flows`). **`GraphExecutor.getGraphModel()`** checks this
  cache first, falls back to the existing lazy path for anything not manifest-listed (ad-hoc/interactive
  playground graphs stay fully dynamic — correctly out of scope).
- Wired into the engine's own test suite (20 graphs incl. `hellojs`) and the shipped
  `examples/minigraph-playground` app (12 tutorials) — all produce byte-identical runtime results to
  before conversion.

### Part 3 — interactive syntax gate + AI-companion guidance (Eric's follow-up ask)

`GraphCommandService`'s `create node`/`update node` handlers (used by both the React playground UI and the
AI companion via `/api/companion/{id}`) now validate `mapping`/`input`/`output`/`for_each` entries **before**
storing them: `->` must be present with non-empty LHS/RHS, and an `f:`-prefixed LHS must reference an
actually-registered plugin — both throw `IllegalArgumentException`, which the existing `handleEvent` catch
already surfaces as `"ERROR: ..."` over the WebSocket (no new error channel needed). Deprecated colon syntax
is auto-converted (not rejected) and a **deprecation notice** is appended to the success response so the
caller — human or AI agent — learns to switch. Also added a "Deprecated syntax" section to
`help graph-data-mapper.md` / `help graph-api-fetcher.md` (proactive guidance — an AI agent asking `help`
learns about this before ever trying the old syntax, not just reactively after being auto-corrected).
