## Summary

**New feature: application log context** (platform-core) — shipped on **PR #128**
(`feature/application-log-context`, commit `7c08679b` off `main`). Began as a design session from
Eric's notes (`~/Desktop/app-log-context.md`) and ran design → spec → implementation → manual
validation → docs → PR in one sprint.

**The gap it closes.** OTel tracing already gives an end-to-end span tree (traceId/spanId across
levels 1/2/3), but the **application log stream** had no correlation — a log line couldn't be tied
back to its trace, and there was no first-class way to carry business context (orderId, user) onto
log output. The traditional answer (Log4j MDC on `ThreadLocal`) is an anti-pattern for a
virtual-thread runtime.

**Design (the load-bearing insight).** Reuse the existing `TraceInfo` mechanism rather than
ThreadLocal: a per-request `LogContext` held in a registry keyed by `Thread.currentThread().threadId()`,
created right after `startTracing` and removed right after `stopTracing` in `WorkerHandler` — the
identical keying the shipped OTel feature already proves correct. The JSON appender runs synchronously
on the worker thread, so it looks the context up by thread id at log time; **nothing is passed through
Log4j**. Rejected `ManagedCache` (TTL/LRU/2000-cap can evict a live context; wrong lifecycle).

**Implementation (org.platformlambda.core.logging):**
- `LogContext` — per-request holder (live `TraceInfo` ref + cid + custom keys), reserved-key set,
  reserved-token resolver (`$utc` = `Utility.date2str(.., true)` UTC).
- `LogContextManager` — `ConcurrentMap<Long,LogContext>` keyed by thread id; sibling of
  `EventEmitter.traces`.
- `LogContextConfig` — loads optional `app-log-context.yaml` via `ConfigReader`; **eager Utility-style
  singleton** (no volatile/DCL — Eric's call); `$token` / `${ENV:default}` / hardcoded-literal entries;
  omits null-valued keys; key order intentionally not preserved (aggregators reorder).
- Hooks: `JsonLogger.getJson` adds the `context` block (covers JsonAppender + CompactAppender);
  `WorkerHandler` register/remove bracket gated on `tracing && isEnabled() && trace.id != null`;
  `PostOffice.updateContext(key,value)` — logging-only sink, distinct from `annotateTrace`, rejects
  reserved keys, no-op when untraced/off.

**Decisions locked with Eric** (via AskUserQuestion + follow-ups): dedicated thread-id registry (not
ManagedCache); `$utc` = per-line log-emit time; create in WorkerHandler + `po.updateContext`;
reserved token `$route`→**`$service`** for vocabulary consistency; **omit** null keys (never `"null"`);
support **hardcoded literals** (e.g. `hello: world`); eager singleton pattern.
