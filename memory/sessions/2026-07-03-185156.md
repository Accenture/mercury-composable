### Code-review fixes (2 medium findings, both confirmed against the code before fixing)

1. **Kafka inbound business cid lost in flow execution.** `KafkaFlowConsumer.toFlowRequest` set only
   `EventEnvelope.setCorrelationId()`, but it dispatches via raw `PostOffice.request()` — whose RPC inbox
   plumbing (`EventEmitter.asyncRequest`) overwrites `.cid` with a temporary reply-routing id *before*
   `EventScriptManager` reads it. And `EventScriptManager.getFlowInstance` reads the business cid from the
   **`correlation_id` header** (`EventScriptManager.CORRELATION_ID`), not that field. So `model.cid` /
   `my_correlation_id` fell back to the routing id. **Fix:** also
   `setHeader(EventScriptManager.CORRELATION_ID, correlationId)` on the forward envelope — mirrors the
   established pattern in `FlowExecutor.request()`. Added unit test
   `KafkaFlowConsumerTest#businessCorrelationIdSurvivesRpcRequestPlumbing` (captures the forward envelope via
   the `invokeFlow` seam; asserts the header carries the cid). See [[kafka-flow-consumer-cid-header]].
2. **`kafka-demo` still referenced the removed DLQ suffix.** The PR replaced global `kafka.flow.dlq.suffix`
   with per-binding `dlq-topic`, but the demo still set the old property, its binding had **no** `dlq-topic`
   (so failures hit the "no dlq-topic → drop-with-ERROR" path = silent loss), and the README documented the
   suffix. **Fix:** removed the stale property, added `dlq-topic: 'demo.inbound.dlq'` to the binding, updated
   README wording. (Repo-wide grep confirmed no other stale `kafka.flow.dlq.suffix` refs outside memory logs.)
