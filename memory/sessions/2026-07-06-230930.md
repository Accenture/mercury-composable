## Fix: flaky forked-JVM teardown (`f67db911`)

Eric flagged a real risk ("embedded kafka can only start once per suite"). Investigation: the concern
doesn't apply to `EmbeddedKafkaTest` (it starts **no** broker — the error/format tests throw or run the
Formatter only; the broker is started exactly twice, both by `MainAppTest`'s dual boot). BUT the run **was**
flaky for a different reason — `mvn clean test` failed intermittently with
`SurefireBooterForkException: Error occurred in starting fork` *after* all 9 tests passed. Root cause:
`MainApp.start()` boots brokers on **non-daemon** KRaft threads and returns without exposing handles
(the `EmbeddedKafka` thread's `run()` returns right after `startServer()`, so the broker handles are
unreachable via thread enumeration), so nothing stopped them but the JVM shutdown hooks — and the async 2nd
broker (8092) could be half-up at exit, hanging/racing its hook → fork teardown failure.

Fix (small production seam, justified — the class already exposed `startServer()`/`shutdown()` for tests):
- `MainApp`: retain started brokers in a package-visible `List<EmbeddedKafka> brokers`.
- `EmbeddedKafka.shutdown()`: package-private → **public** (embedding process / test can stop it directly).
- `MainAppTest`: capture `app.brokers`, `awaitBroker(9092)` **and** `awaitBroker(8092)` (AdminClient
  describeCluster, no Thread.sleep) so both are fully up, then shut both down explicitly in `@AfterAll`.
- Verified: 3 consecutive clean runs green, zero fork errors. Coverage held: line 86.8%, instruction 88.1%,
  branch 77.3% (Sonar blended ~84.7%).

Lesson: booting a fire-and-forget server launcher (MainApp) in a test leaves non-daemon threads that break
surefire fork teardown unless you capture handles and shut down explicitly — don't rely on JVM hooks.
