Diagnosed and fixed the SECOND field Jenkins failure of the twin-kafka module — the one that

The secondary adapter **crashed at startup** (15:01:04, seconds into the run); the 90s poll then
waited for a route that could never register and failed at `boot:104` after 94.81s. The deadline
was never the problem — **PR #178's "slow executor" diagnosis treated a symptom.** The first
Jenkins failure almost certainly had this same root cause (same assert, same shape).

## Root cause (three interacting facts)

1. **`AppConfigReader` resolves `${VAR}` references ONCE**, when the singleton first loads
   (`ConfigReader.load` → `resolveReferences()` bakes the values in).
2. **Surefire runs all of a module's test classes in one JVM** (default `forkCount=1`,
   `reuseForks=true`), and its **default run order is filesystem-dependent** — so class order
   differs across CI environments.
3. `SecondaryKafkaHealthCheckTest` (added in 4.8.4) touches `AppConfigReader.getInstance()` in its
   constructor chain (`resolveDurationMs`). On the field Jenkins agent it ran BEFORE
   `TwinKafkaBridgeTest`, so the singleton loaded while `SECONDARY_SCHEMA_REGISTRY_URL` was still
   unset — freezing `secondary.schema.registry.url` to blank. The bridge test then set the env-style
   property too late. GitHub CI's ordering happened to run the bridge test first, masking the bug.

**Reproduced locally**: `mvn test -f system/twin-kafka/pom.xml -Dsurefire.runOrder=alphabetical`
(S before T) fails identically to the field — `boot:104 secondary Kafka flow adapter should have
started`, 93s. Confirms order-dependence as the mechanism.

## Fix (branch `fix/twin-kafka-test-order-independence`, commit `a4624120`)

1. **Inject via the EXACT config key**: `System.setProperty("secondary.schema.registry.url",
   registryB.baseUrl())` — `ConfigReader.get()` checks `System.getProperty(<exact key>)` live on
   EVERY read, so it is immune to initialization order. This is the same pattern minimalist-kafka's
   `KafkaFlowAdapterTest` already used; the bridge test had diverged from it. The
   `KAFKA_BOOTSTRAP_SERVERS`-style env references remain safe because client templates are loaded
   fresh (new ConfigReader) after boot() sets them, and template resolution consults the base
   config's live system-property check.
2. **Pinned `runOrder=alphabetical`** in twin-kafka's surefire config so every environment
   (GitHub, field Jenkins, local) runs the same adversarial order — a regression of this class now
   fails everywhere, not just in the field.
3. **Boot comment** now tells the next reader to check for an AppStarter "Unable to start" ERROR
   before blaming a slow executor — a crashed `@MainApplication` never registers its adapter and no
   deadline extension fixes that.

Verified: alphabetical order green in 30s (was 93s fail); plain `mvn test` (now pinned) green 30.8s.
