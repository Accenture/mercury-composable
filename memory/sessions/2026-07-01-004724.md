## Summary

**New feature: schema-id resolution from subject + version for `simple.kafka.notification`** — designed
→ spec → implemented → manually validated end-to-end → shipped on **PR #129** (merged to `main` as
`9995da3a`). Fulfils the planned thread [[thread-kafka-schemaid-from-name]].

**The change (contract pivot).** `simple.kafka.notification` was **id-driven** (`schema-id`/`schema-type`
headers → `GET /schemas/ids/{id}`). It is now **subject-driven**: the caller supplies a registry
**`subject`** (+ optional **`version`**, default `latest`); the producer resolves subject/version → global
id + type and serializes through the unchanged `use.schema.id` path. **The old id-driven headers were
removed outright** (no backward-compat shim — 4.5.0 of minimalist-kafka is unreleased, confirmed by Eric).
It is the **single** contract, not an added option — deliberately, so developers aren't faced with the
choice that confuses them.

**Key design decisions (with Eric):**
- **Subject/version only; drop the 3 Confluent naming strategies.** Strategies only matter when a producer
  *derives* a subject from `(topic, record)` and auto-registers; since the caller *names* the subject,
  there's nothing to derive. This is the confusion being removed.
- **Type resolved authoritatively** from `getSchemaById(id).schemaType()` (not a possibly-absent metadata
  field), which also warms the id→schema cache.
- **Caching by mutability → 2 caches** (Eric trimmed a 3rd): pinned **numeric versions** (immutable) in a
  dedicated long-TTL (10d) bounded (3000) `ManagedCache`; **`latest`** (mutable) shares the existing 30m
  id→schema cache under a `"latest/"`-namespaced key (can't collide with digit-only id keys). Eric's call:
  collapse latest into the 30m cache for simplicity rather than a dedicated latest cache.
- **Wire + consumer unchanged** — global id still on the wire; only the producer's *input* moved.
- **Mock seeding model** stays "drop `<id>.json` + (re)start" (dev tool, not a product); runtime adds go
  through the register API. Eric: don't grow the standalone registry into a real product.

**Scope of change:** `KafkaHeaders` (SUBJECT/VERSION replace SCHEMA_ID/SCHEMA_TYPE);
`ManagedCacheSchemaRegistryClient.resolve(subject,version)` + `ResolvedSchema`; `SchemaCodec` (version
cache + `resolve` facade); `SimpleKafkaNotification.encode` (subject-driven; unknown subject / registry
down → fail loud, never silent plaintext). Standalone mock: new `GET /subjects/{subject}/versions/{version}`
(latest|int, Confluent shape, 40401/40402), `SchemaStore` in-memory `subject→version→id` index rebuilt on
boot from per-id files (each now records subject+version), `register(subject,...)`, + a module **README**
documenting every endpoint. `EmbeddedSchemaRegistry` got the same endpoint + version tracking.

**CSFLE: ready, not wired (next iteration).** Confluent CSFLE lives in the schema's ruleSet (subject-centric),
so this subject/version base is the right foundation, BUT minimalist-kafka's serdes pass no rule executors /
KMS today — a CSFLE-ruled schema would serialize in **plaintext**. Out of scope here; the kafka-flow-adapter
guide carries a safety caveat (don't produce to CSFLE-protected topics until it lands). See
[[thread-csfle-field-encryption]] — the agreed next feature.
