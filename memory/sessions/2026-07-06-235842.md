## redis-standalone 0% → ~89.3% (`9eb7873a`) — the `setTestMode` trick (Eric's hint)

Had **no tests**. A thin launcher; its failure paths were unreachable (`EmbeddedRedis` fails fast via
`System.exit(-1)`; `redisServer.stop()` doesn't throw in normal use). Booting via `AutoStart` in a Redis
test **hangs** (no REST server → platform doesn't terminate; a `MainApp.main()` attempt hit the 5-min
timeout), so `MainApp.main()` stays uncovered.
- **`RedisStandaloneTest`** (new, example pkg): boots via `new MainApp().start()` (handle retained in a
  MainApp `redis` field) and round-trips PING/SET/GET over a **raw RESP socket** (no Redis client dep);
  test `redis.port=6399` avoids a local 6379; stopped via the handle.
- **`EmbeddedRedis`**: extracted a package-visible `startServer()` (matches sibling `EmbeddedKafka`) and
  added **`setTestMode(boolean)`** — when on, `startServer()`/`stop()` emulate an `IOException` and `start()`
  skips `System.exit` (`if (!testMode)`), making the catch/guard branches reachable from a unit test without
  killing the JVM. `setTestMode` is itself covered by the test call.
- **`EmbeddedRedisTest`** (new, system pkg — reaches the package-visible seams): exercises those failure
  branches + the stop-before-start no-op.
- Result: blended ~89.3% (line 91.3%, instruction 92.8%, branch 80.0%); EmbeddedRedis 94%; 5 tests, 2 runs.
- Key lesson: **unreachable fail-fast/`System.exit` branches → add a package-visible `setTestMode` that
  emulates the exception**; test the method directly (not via the `System.exit` wrapper). Applies to any
  standalone launcher here.
