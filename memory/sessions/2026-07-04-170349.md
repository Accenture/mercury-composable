### Scope decisions (Eric, via AskUserQuestion)

Surveyed the whole codebase first (54 Java files touched `correlationId`/`CorrelationId`/`CORRELATION_ID`)
before touching anything, since renaming public API / wire contracts on a v4.5.0 released library is a
real breaking-change decision. Two scope questions:

1. **Exclude `EventEnvelope.cid`/`getCorrelationId()`/`setCorrelationId()`** (and the identical look-alike
   pattern in `minigraph-playground-engine`'s `GraphInstance` and `mini-scheduler`'s `JobExecutor`) — it's a
   much older, generic RPC reply-matching primitive used at 100+ call sites for plain request/reply,
   predating and unrelated to the business-cid feature. Left as-is; added a doc note instead of renaming.
2. **Java identifiers only** — rename private fields/local vars/internal constants to
   `businessCorrelationId`/`BUSINESS_CORRELATION_ID`, but keep the public method `getMyCorrelationId()`
   and the wire header strings (`"my_correlation_id"`, `"correlation_id"`) unchanged. Zero breaking impact
   on consumers or deployed configs.
