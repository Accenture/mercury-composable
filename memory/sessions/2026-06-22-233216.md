## Decisions Made

- **Stage term = "protocol boundary"** (not "event boundary"). Rationale, recorded inline in
  the `request-pipeline-model` fact so it isn't reverted: (1) precision — at the boundary a
  request is still a protocol message; it only becomes an `EventEnvelope` one stage later at the
  flow adapter, so "event boundary" implied events exist too early; (2) code-groundability — no
  artifact is named "event/protocol boundary", but "protocol boundary" reads as a description,
  not a phantom class an AI would hunt for, and the concrete anchors (REST automation, flow
  adapter, `http.flow.adapter`) carry the weight; (3) "entry point" was rejected — collides with
  Mercury's `EntryPoint` / `@MainApplication`.
