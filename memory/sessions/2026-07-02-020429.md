### The investigation (this is the valuable part — a field insight overturned the original plan)
Eric reported that in the field, CSFLE is normally just **delegated** to the Confluent serdes via config —
no framework code required — and asked whether minimalist-kafka could do the same instead of my originally
planned "attach the ruleSet ourselves" approach (Option A from the prior session, which added a `resolve()` /
`ResolvedSchema` change). Decompiling the actual Confluent 8.2.0/8.3.0 jars settled it:

- `executeRules(RuleMode.WRITE)` fires in **every** serialize branch including `use.schema.id` (the branch
  minimalist-kafka already uses) — rule execution is not gated on `use.latest.with.metadata`.
- `SchemaString` (the `GET /schemas/ids/{id}` response, i.e. our existing `getSchemaById` path) **already**
  carries `getMetadata()`/`getRuleSet()`, and `getSchemaById` parses them onto the returned `ParsedSchema`.

**Conclusion: pure delegation works. No `resolve()`/`ResolvedSchema`/`SchemaStore` change needed at all** —
just add the encryption executor + a KMS driver jar, and let CSFLE config reach the serializer/deserializer.
This superseded the prior session's Option A/B design entirely (spec bumped to v2.0, then v3.0 after the
open-questions round).

**A second finding, while building the verification spike:** the encryption executor needs Confluent's
**DEK-registry** subsystem (`DekRegistryClient.getOrCreateKek`/`getKek`), which neither `EmbeddedSchemaRegistry`
nor the standalone mock implement. Turned out not to matter: `EncryptionExecutor.configure()` builds its
`DekRegistryClient` from the **same `schema.registry.url`** value our schema client already uses, and
`DekRegistryClientFactory` honors the identical `mock://<scope>` test convention as
`SchemaRegistryClientFactory` (both are Confluent test utilities, same `MOCK_URL_PREFIX`/scope-map pattern).
So a single `mock://<scope>` URL backs both clients — **zero HTTP mocking, zero EmbeddedSchemaRegistry
changes** for local verification. This is why task "extend EmbeddedSchemaRegistry" (original task #13) was
superseded/dropped.

### The open-questions round — one real correction surfaced
Eric answered 5 open questions from the spec. #1/#3/#4/#5 confirmed the existing plan (AWS-only-by-default in
pom.xml; fixed test secret is deterministic enough; field KMS validation stays manual in Eric's environment,
not CI; DEK-registry seam already resolved). **#2 surfaced a genuine mistake**, caught by re-verifying via
decompilation rather than taking the answer at face value:

Eric said CSFLE config must bind **per-subject** in field installations. Checking
`FieldEncryptionExecutorTransform.getKekName`/etc. confirmed `RuleContext.getParameter` resolves
`encrypt.kek.name`/`encrypt.kms.type`/`encrypt.kms.key.id` from the **registered rule's own params** (or the
schema's `Metadata`) — **never** from the executor-level config map our `schema.registry.serde.*`
pass-through feeds. So per-subject binding was **already correct** in the design (different subjects can use
different KEKs/vendors with zero code change) — but the guide's example and both new tests had **redundantly
duplicated** those 3 keys into the serde-level config too, which incorrectly implied a single global KEK.
Fixed: guide now states the split explicitly (rule params = per-subject/schema-owned, set by whoever governs
schemas; `schema.registry.serde.*` = reserved for the KMS **driver's own** credentials only, e.g. AWS's
`access.key.id`/`secret.access.key` confirmed via `AwsKmsDriver`'s own constants); `SchemaCodecCsfleConfigTest`
now only puts the local-KMS driver's `secret` in `extraSerdeConfig`. Re-ran both test files after the
trim — identical pass, confirming those 3 keys were never actually read from that config map.
