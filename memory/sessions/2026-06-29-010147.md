## Part 2 — minimalist-kafka Schema Registry support, JSON Schema (commit `7caa3434`)
Designed with Eric (plan approved). Use Confluent's **own serializers as a library** (no reinvented wire
format), keeping the String/byte[] transport so DLQ + trace still work on raw bytes.
- **Producer is id-driven** (Eric's simplification): `simple.kafka.notification` gains optional
  `schema-id`/`schema-type` headers; schemas are pre-registered administratively, so the serializer only
  does `GET /schemas/ids/{id}` (`use.schema.id` + `auto.register.schemas=false`). No subject/strategy
  needed → all 3 Confluent strategies supported by construction; many record types per topic.
- **Consumer** opt-in `schema.enabled`: read magic id → registered `schemaType` → matching deserializer →
  Map; decode failure dead-letters the raw record.
- Codec layer (`mini.kafka.schema`): `SchemaCodec`, `FileCachedSchemaRegistryClient` (on-disk schema cache
  by id, TTL — serves the serde `getSchemaById` hot path), `SchemaType`.
- Dep: `io.confluent:kafka-json-schema-serializer:8.2.0` (CP 8.2 ↔ Kafka 4.2), transitive kafka-clients
  excluded.
- 35 tests, 85% coverage gate met. Self-contained in-JVM `EmbeddedSchemaRegistry` (JDK HttpServer) backs
  the tests; one test decodes a message produced by a **stock** `KafkaJsonSchemaSerializer` (real interop).
- The real serde only called `POST /subjects/{subject}/versions` + `GET /schemas/ids/{id}` — both already
  in the standalone mock, so it is already Confluent-compatible for this phase.
