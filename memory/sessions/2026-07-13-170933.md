## 3. Declarative async.http.request trace investigation (`fix/declarative-http-trace`, `618ba87e`)

Field report (via devops): traceparent propagates programmatically but NOT declaratively
(flow task `process: 'async.http.request'`) on 4.7.x; same config "worked" on 4.4.11.
Screenshots showed the field flow: traced ingress → input-mapped async.http.request task
with `execution: sink`. Built two regression tests (echo endpoint + new TraceRecorder
fixture for the sink shape, since sink output never reaches the flow) and ran them on
BOTH current main and a v4.7.1 worktree with the field's exact traceparent:
**ALL PASS on both versions — the declarative path stamps X-Trace-Id + traceparent
correctly, including the sink-after-response shape.** Not a framework defect; the
engine's task dispatch (`new PostOffice(SERVICE_NAME, flowInstance.getTraceId(), ...)`)
is byte-identical between 4.7.1 and HEAD, and AsyncHttpClient's stamping predates 4.6.3.
Remaining field suspects (same family as the morning's diagnosis): downstream endpoint
missing `tracing: true`; a gateway stripping trace headers between the apps; or the
validation watching X-Correlation-Id (the retired 4.4.x conflation). Tests committed as
permanent coverage — the gap Eric named was a COVERAGE gap, now closed.
Also learned: a flow must have ≥1 `end` task — a sink can only terminate a side branch
(fixture used response → parallel fan-out → sink + end).
