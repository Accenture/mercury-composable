## Investigation: OAuth 2.0 for Schema Registry in minimalist-kafka (v4.6.3 field loose ends)

Field context: 4.6.3 deployed to test sync-over-async + minimalist-kafka. Two loose ends:
(1) Confluent/Kafka OAuth token URLs must be on a JVM allow-list; (2) SR HTTP calls need a
bearer token from a client-credentials token endpoint (Azure AD style).

### Verified against the actual jars (kafka-clients 4.2.0, kafka-schema-registry-client 8.3.0)

- **No custom OAuth client needed.** Confluent SR client has full client-credentials support:
  `bearer.auth.credentials.source=OAUTHBEARER` + `bearer.auth.issuer.endpoint.url` /
  `bearer.auth.client.id` / `bearer.auth.client.secret` / optional `bearer.auth.scope`.
  Token fetch, JWT validation and caching w/ expiry buffer are built in
  (`CachedOauthTokenRetriever`, `bearer.auth.cache.expiry.buffer.seconds` default 300s).
- `bearer.auth.logical.cluster` / `bearer.auth.identity.pool.id` are OPTIONAL
  (`validateString(key, false)` in `OauthCredentialProvider.configure`) → plain Azure AD
  endpoints work without Confluent Cloud extras. Other sources: `STATIC_TOKEN`,
  `SASL_OAUTHBEARER_INHERIT` (reuse kafka client SASL config), `CUSTOM`.
- **Allow-list applies to BOTH token URLs**: Kafka's
  `ConfigurationUtils.validateUrl` ends with `throwIfURLIsNotAllowed` reading system property
  `org.apache.kafka.sasl.oauthbearer.allowed.urls` (still a JVM property in kafka 4.2.0, not a
  client config), and Confluent's `OauthCredentialProvider.getTokenRetriever` calls that same
  `validateUrl` on `bearer.auth.issuer.endpoint.url`. Matches Eric's field workaround.
- Bearer auth activates via the existing `configs` map: `CachedSchemaRegistryClient` →
  `RestService.configure` → `BearerAuthCredentialProviderFactory`. `SchemaCodec.fromConfig`
  currently passes only registry URL + negative-cache pins → the GAP is config plumbing.

### Design (pending Eric's review)

1. Pass-through prefix `schema.registry.client.*` → srConfig map (mirrors the existing
   `schema.registry.serde.*` pattern in SchemaCodec) — carries `bearer.auth.*` (and
   `basic.auth.*`, ssl) to the registry REST client; env-var substitution keeps secrets out
   of files.
2. Auto-register OAuth token URLs on the JVM allow-list (merge+dedupe with any existing
   value, never clobber): `sasl.oauthbearer.token.endpoint.url` from producer/consumer
   templates at KafkaClientConfig load; `bearer.auth.issuer.endpoint.url` at SchemaCodec
   build. Registration must precede client construction (it does at those points).
3. Tests: schema-registry-standalone mock gains an optional token endpoint + Bearer
   enforcement; unit tests for allow-list merge helper + pass-through extraction +
   end-to-end authenticated resolve. Transport OAuth vs embedded broker NOT feasible
   (PLAINTEXT) — transport side stays config-only + allow-list automation.
