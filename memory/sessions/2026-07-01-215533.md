### Investigation (dependency CVEs)
Root cause for all three flagged modules: each has `spring-boot-starter-parent` as its own direct parent,
and Maven resolves every transitive `netty:*`/`jackson-databind` version using **the consuming module's
own** `netty.version`/`jackson-2-bom.version` properties тАФ not whatever `platform-core` (or any other
dependency) declares internally. `minimalist-kafka` and `sync-over-async` had **no** `netty.version`
override at all (silently inheriting the parent's older default); `reactive-postgres` had one, but it was
stale (`4.2.13.Final` тАФ 3 more netty CVEs were fixed only in `4.2.15.Final`, confirmed via OSV.dev: IPv6
filter bypass, TLS hostname-verification bypass, SNI 16 MiB pre-allocation, HTTP/3 memory exhaustion).
`wire-runtime-jvm` (Confluent's protobuf dependency) has **no supported fix at all** тАФ discontinued
artifact; only fix is Confluent switching to the renamed `wire-runtime` coordinate upstream. One
jackson-databind CVE (`@JsonIgnoreProperties` bypass) also has no fix yet in the 2.x line. Both are
tracked as "wait on upstream," not something a pom edit resolves.

The two SAST findings (AppConfigReader.java deserialization, HttpRouter.java CRLF injection) were checked
empirically against the actual libraries in use rather than trusting the taint pattern: SnakeYAML 2.5's
default `Yaml()` constructor rejected a classic RCE gadget payload (`ComposerException: Global tag is not
allowed`), and Netty 4.2.12's `DefaultHttpHeaders` (what Vert.x's `HttpServerResponse.putHeader` uses)
rejected an embedded `\r\n` (`IllegalArgumentException: Validation failed`). Both already mitigated by
library defaults; no code change needed.

### Applied (per Eric's explicit hints, repo-wide)
- **`spring-boot-starter-parent`**: `4.1.0` for every module except `system/rest-spring-3` +
  `examples/rest-spring-3-example` (the Spring Boot 3.x line) тЖТ `3.5.16`. 28 pom.xml files.
- **`netty.version` property тЖТ `4.2.15.Final`** everywhere it already existed (26 files), **added** it to
  the 2 files that never had it (`system/minimalist-kafka`, `extensions/sync-over-async` тАФ the actual root
  cause of the two Snyk findings on those modules).
- **`tomcat.version`**: `11.0.23` everywhere except `rest-spring-3`/`rest-spring-3-example` тЖТ `10.1.56`.
- **`vertx-core` тЖТ `5.1.3`** тАФ single declaration point (`system/platform-core/pom.xml`), everything else
  inherits it transitively.
- **`xsi:schemaLocation` тЖТ `https://`** тАФ fixed the literal `maven-v4_0_0.xsd` URL Eric flagged (21 files)
  and, same class of IDE warning, the alternate `xsd/maven-4.0.0.xsd` form too (7 files not explicitly
  named but the same fix).

Verified along the way (not assumed): Spring Boot 4.1.0's own BOM default already includes
`jackson-2-bom.version=2.21.4`, so the jackson-databind fix landed "for free" via the parent bump alone
(fixes 2 of the 3 flagged CVEs). Spring Boot 4.1.0 also silently bumped `lettuce-core` `6.8.2.RELEASE` тЖТ
`7.5.2.RELEASE` (its own managed default, a major-version jump neither of us asked for) тАФ ran the full
sync-over-async suite specifically because of this and it's fully green (32/32, incl. the Redis Pub/Sub
coordinator tests).
