Implemented **consumer partition pinning** for the Kafka Flow Adapter — an opt-in per-binding feature
(post-MVP item in `thread-redis-kafka-rpc`).

## Change
- `kafka-flow-adapter.yaml` consumer entry gains an optional `partition:` field. When present,
  `KafkaFlowConsumer` manually **`assign`s** that single `TopicPartition` instead of group-managed
  **`subscribe`** — bypassing consumer-group rebalancing (the pinned consumer reads exactly that partition;
  offsets still commit under the configured group). Absent → unchanged subscribe behaviour.
- `KafkaFlowAdapter.parsePartition(Object)` — parses the optional value (non-negative int, else
  `IllegalArgumentException`); `${ENV_VAR:default}`-substitutable like the rest of the YAML, so each pod can
  pin a distinct partition (`partition: ${POD_PARTITION}`).
- `KafkaFlowConsumer`: new nullable `partition` ctor param + `subscribeOrAssign(consumer)` seam (called by
  the poll loop). Javadoc documents the deployment-model trade-off (operator owns it; no group load-balancing
  when pinned).
