## Summary

P5 (off-loop dispatch) of the ElasticQueue file-FIFO effort — the principled fix for the P2 552 ms
OS-flush outlier. Branch `feature/elastic-queue-file-fifo`, commit `e8e5cdee`. See
[[thread-elastic-queue-bdb-to-file]] and [[elastic-queue-file-fifo-plan]]; relates [[virtual-threads-rpc]].

### Design constraint that shaped it
The spec's P5 sketch ("keep fast path on loop; move only spill to a VT") is impractical because
`ElasticStore` is single-threaded per route — its read/write/counters + the buffering state machine all
converge onto one thread. So the realistic models were **full-actor** (per-route VT owns the state machine +
store) vs an intricate, racy hybrid. Eric chose **A (full-actor) behind a default-off flag**.

### Implemented
`elastic.queue.dispatch = loop | vthread` (default `loop`). In `vthread`: the Vert.x consumer only enqueues
(non-blocking) to a per-route mailbox; a per-route **virtual thread** drains it and runs the whole
`ServiceQueue` state machine + blocking spill I/O. One VT per route ⇒ single-threaded-per-route ordering
preserved; the drain loop catches exceptions so a transient spill error can't kill a route's dispatch;
`stop()` interrupts + joins the VT before destroying the elastic queue. Same de-risking pattern as `bdb|file`
(default = today's inline behaviour; opt-in, pipeline-flippable). `stopped` made volatile (cross-thread read).

### Verification
- **Functional parity: full platform-core 381 tests green in BOTH modes** (loop + vthread) — the dispatch
  change is transparent across RPC/streaming/callbacks/REST/back-pressure.
- **`DispatchBenchmark`** (fire-and-forget throughput, gated `-Dbench.run=true`): loop **83k** vs vthread
  **98k events/s (+18% under load)** — because spill I/O is off the loop, so the loop keeps enqueuing. No
  hot-path regression; an improvement under load. (An RPC-latency variant was abandoned — reply routing from a
  plain test thread was flaky; fire-and-forget throughput is the robust measure.)

**Result:** the P2 552 ms OS-flush stall is now **harmless** in `vthread` mode — it parks the VT's carrier,
never the shared event loop.

### Measured mailbox overhead (Eric asked to see it) — commit `f049111a`
Added `DispatchBenchmark#fastPathDispatchOverhead`: in-flight=1 sequential sends + 5 spare-ready workers ⇒ a
worker is always ready ⇒ direct dispatch, **no spill** — so loop-vs-vthread isolates the ServiceQueue mailbox
hand-off. Send→worker dispatch latency (N=200k): **loop p50 11.9µs / p99 32.8µs vs vthread p50 20.6µs / p99
41.1µs → ~5–9 µs per dispatch** (LinkedBlockingQueue offer→take + VT wake). Small absolute cost on the idle
fast path (which has headroom anyway); under load vthread is +18% and the 552 ms stall is neutralized —
a sound trade for a back-pressure buffer.
