Refactored the `sync-over-async` extension from imperative code into the composable (Event Script Level 2)
pattern, and closed a real loose end in the design — publish fail-fast. Eric drove the design; this was an
iterative collaboration over several turns (composable flow → reactive publish → promotion → docs).

## Changed
- **Composable flows.** `sync-to-async.yml` is now 3 declarative tasks — `sync.prepare` → `simple.kafka.
  notification` (publish) → `sync.await` (block for the async reply) — replacing the single imperative
  `test.endpoint`. `system-of-record.yml` uses a `decision` task (echo + decide → publish reply, or `no.op`
  drop for the timeout path). The only Kafka publishing now goes through the reusable `simple.kafka.
  notification` building block.
- **Fail-fast publish (reactive).** `KafkaRequestPublisher.publish` → `Mono<Void>` (eager send, completes on
  broker ack / errors on failure); `SimpleKafkaNotification` returns it. The worker subscribes and defers
  completion until ack, so the facade rejects on publish failure instead of waiting out the timeout. Backend-
  unreachable 5xx → **503** in `sync.error.handler`. Verified platform-core handles `Mono` transparently
  (`WorkerHandler.handleMonoResponse`).
- **Promotion.** `SyncPrepareTask` / `SyncAwaitTask` promoted to main, `instances=250` (virtual threads, so
  250 concurrent blocking awaits are cheap — the ADR-0002 payoff). Eric then moved them to a new
  `org.platformlambda.tasks` package (readability convention). Added `awaitResponse(cid, timeout)` overload
  with a Redis-read fallback (race-safe across the split prepare/await tasks).
- **Decision-task doc fix.** The engine routes boolean `true → next[0]` (first), `false → next[1]` (second);
  integer is 1-based, multi-way (a switch). `syntax.md` was correct; the *derived* docs (flow-schema-
  reference, ai-agent-guide, flow-grammar, event-script-flow.json) had it inverted — all corrected.
