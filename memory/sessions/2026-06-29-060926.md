## Review-driven hardening of the Kafka flow adapter (commit `1f9f8cc8`)
- **Flow ttl is the processing deadline** (Kafka is async — no inherent request timeout). Dropped
  `kafka.flow.timeout.ms`; the consumer waits `Flows.getFlow(flowId).ttl` and the adapter fail-fasts at
  startup on an unknown flow.
- **Flow success = HTTP 200**; any other status (or a thrown exception/timeout) → retry → DLQ. Previously a
  non-200 reply was silently committed.
- **DLQ-write failure → loud ERROR + commit (drop the message)** instead of redeliver-forever — avoids a
  recovery storm. Documented data-loss caveat + the future "alternative path" (persist to a store). New
  `kafka.dlq.timeout.ms` bounds the confirmed DLQ write.
- **Envelope serialize**: `JsonSchemaUtils.envelope(schema, value)` with the pre-registered schema
  (`use.schema.id`) instead of deriving from `Map<String,Object>` — kills the kjetland "Unable to process
  java.lang.Object" WARN + wasted work.
- **Codec cache cleared at startup** (`schema.registry.cache.dir`): it's rebuildable, so no stale entry
  carries across runs.
- SonarQube: split `routeToFlow` + the adapter constructor (cognitive complexity); suppressed the
  false-positive try-with-resources on the shared publisher; test cleanups (assertNotNull, restricted
  `record` identifier, `@SuppressWarnings("resource")`).
