### Finding — the ElasticQueue two-tier throughput cliff (laptop, file store, 200k ops)
| scenario | throughput | mean | note |
|---|--|--|--|
| RPC 1→50 | 21k | 0.047 ms | baseline, in-flight=1 |
| RPC 50→50 | **165k** | 0.30 ms | **in-memory tier** (queue ≤20) |
| Callback 50→50 paced | 39k | **0.47 ms** | healthy async, no spill |
| RPC 100→50 | 75k | 1.33 ms | **disk-spill tier** (queue ~50 >20) |
| Callback flood→50 | 78k | 25 ms | disk-spill tier |

**Crossing the 20-event memory buffer roughly halves throughput:** in-memory ≈165k ops/s vs disk-spill
≈75–78k ops/s. Both spill scenarios (RPC 100→50 and the callback flood) land at the same ~75–78k ceiling —
strong evidence it's the spill path, not the workload. **Correction to the going-in assumption:** RPC 100→50
(2:1 oversubscription) does *not* stay out of back-pressure — ~50 requests queue (>20), so it spills; latency
rises ~4.6× (not the naive 2×) and throughput drops to the spill ceiling. Eric's call: **categorize it as
back-pressure** (it demonstrates back-pressure via oversubscription, complementing the flood). Paced async is
sub-ms (0.47) vs the flood's 25 ms — ~53× — showing healthy vs overloaded on the same 50 consumers. **0
failures / 0 loss everywhere** — back-pressure degrades predictably (Little's law), never collapses.
**Takeaway:** well-architected = keep in-flight ≤ consumers to stay in the fast in-memory tier; scale past a
single route's ceiling horizontally (more routes), per the earlier per-route-dispatch finding.
