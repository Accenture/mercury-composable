## Summary

**Removed Protobuf support from `minimalist-kafka`'s Schema Registry integration, end-to-end, before its
first release.** Snyk's pipeline gate is a hard must-pass for publishing to client Artifactories, and it
flags `com.squareup.wire:wire-runtime-jvm:5.5.0` (CVE-2026-45799 / GHSA-7xpr-hc2w-34m9 — a crafted 10-byte
payload crashes any Wire-decoding consumer) with no fix available: the coordinate is discontinued, and
Confluent's `kafka-protobuf-provider` hasn't adopted the renamed `wire-runtime` replacement even as of
8.3.0 (checked directly against Confluent's own POM in a prior session — `wire.version` unchanged 8.2.0→8.3.0).
Getting a security exemption isn't viable, so Eric decided (confirmed via discussion, including an industry
check on Avro/Protobuf/JSON schema-registry market share — no hard % found but strong 2025-2026 qualitative
consensus that Avro dominates Schema-Registry-backed Kafka specifically, Protobuf's strength is a different
ecosystem, gRPC/polyglot) to drop Protobuf rather than gamble on an unverified major-version substitution
(`wire-runtime-jvm` → `wire-runtime` 6.3.0+) under a hard release-gate deadline. Explicit requirement: state
the rationale and limitation in user-facing docs (not silently drop it) — an honesty/objectivity principle
Eric named directly.

### Code removed (system/minimalist-kafka)
- Deleted `ProtobufSchemaSerde.java`, `ProtobufConversions.java`, `ProtobufConversionsTest.java`.
- `SchemaCodec.java`: removed the `ProtobufSchemaProvider` import/registration and the
  `serdes.put(SchemaType.PROTOBUF, ...)` line; added class-Javadoc explaining why. **Deliberately did not**
  remove `SchemaType.PROTOBUF` from the enum — `SchemaCodec`'s existing `serde()` helper already throws a
  clean `UnsupportedOperationException` for an unregistered type, so keeping the enum value means a
  misconfigured attempt (producer resolves a subject to a Protobuf-typed schema, or a caller passes
  `SchemaType.PROTOBUF` directly) fails clearly instead of an NPE or an enum-parse error. Updated
  `SchemaType.PROTOBUF`'s own Javadoc with the full rationale — it's the first place a future developer
  looks when they see a recognized-but-unwired enum value.
- `pom.xml`: removed the `io.confluent:kafka-protobuf-serializer` dependency block entirely (required, not
  optional — Snyk scans the resolved dependency tree, so leaving the vulnerable jar on the classpath while
  just not calling it would not have satisfied the gate).
- Tests: removed the two Protobuf round-trip tests from `SchemaCodecTest.java` and the one from
  `KafkaFlowAdapterTest.java`; added a new regression test (`protobufIsRecognizedButNotSupported`) asserting
  the `UnsupportedOperationException` contract holds.

`node/create-topics.js`. JSON/Avro legs untouched.
