## Summary

Built the **config-driven init** for sync-over-async (post-MVP item "Redis coordinator config-driven init").
Before this, the module's `main` was only the engine; the `RedisClient` + coordinator were wired in test code
(hardcoded `redis://127.0.0.1`, `pod-mvp`). Now there is a production bootstrap mirroring minimalist-kafka's
`KafkaFlowAutoStart`/`KafkaRuntime`.

Decisions (Eric): **discrete Redis properties** (over a single URI); **full config-driven init** scope.

### New (main)
- `RedisConfig` (record) — discrete startup params `redis.host`/`redis.port`/`redis.password`/`redis.ssl`/
  `redis.database`/`redis.timeout.ms` → `from(ConfigBase)` + `toUri()` (Lettuce `RedisURI`).
- `SyncOverAsyncConfig.from(ConfigBase)` — engine tunables `sync.return.channel.prefix`/`sync.route.ttl.seconds`/
  `sync.response.ttl.seconds`/`sync.max.pending.requests` (keeps `defaults()`).
- `SyncRuntime` (production holder) — coordinator + client + `shutdown()` (coordinator must NOT own the shared
  client; the engine tests share one client across two coordinators, so the holder owns client teardown).
- `SyncOverAsyncAutoStart` (`@MainApplication`) — **opt-in** via `sync.over.async.enabled=true` (default false,
  because the coordinator eagerly connects to Redis); builds client + coordinator keyed by
  `Platform.getInstance().getOrigin()`, starts it, stores in `SyncRuntime`.

### Changed (test)
- MVP `RestFlowMvpTest` now boots through the production autostart (embedded Redis injected via the `redis.port`
  system property; `sync.over.async.enabled=true` in test application.properties) — removed the hand-wired
  coordinator. Fixtures `TestEndpoint`/`SoaReplyTask` use `org.platformlambda.sync.SyncRuntime.coordinator()`
  (FQN — same-simple-name as the test-only topic holder). Test `SyncRuntime` keeps only the topic constants.
- New unit tests `RedisConfigTest` (3), `SyncOverAsyncConfigTest.from` (2), backed by a small in-memory
  `ConfigBase` double `MapConfig`.

### Notes / gotchas
- `ConfigBase` is in `org.platformlambda.core.util.common` (not `...util`); `ConfigReader.get()` does
  dotted-path traversal, so flat dotted map keys do NOT resolve via `ConfigReader.load(Map)` — hence the
  `MapConfig` double for unit tests.
- The autostart-created `RedisClient` lives for the app lifetime (JVM reclaims on exit); `SyncRuntime.shutdown()`
  is the clean teardown (used by the test; a production shutdown hook could call it).

### Validation

  proves the full REST → Kafka → SoR → Kafka → Redis return-route round-trip + 408 timeout through the
  production autostart.
