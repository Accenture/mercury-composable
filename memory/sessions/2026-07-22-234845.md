1. **Logger recursion fix (`b38af0a3`).** Eric reported "Recursive call to appender JsonLogger"
   on composable-example startup. Root cause: `LogContextConfig` is a lazily-referenced singleton
   whose constructor logs; when the first reference happens *inside* a JSON appender that log4j2
   is still initializing (during `Configurator.reconfigure`), the INFO line re-enters the
   half-built appender. Fix: `AppStarter.reConfigLogger` now touches
   `LogContextConfig.getInstance()` **before** reconfiguring log4j2, so the config's own log
   lines ride the plain text logger.

2. **RPC span lineage (`04e5618f`).** Eric's declarative event-over-http demo exposed that
   `round_trip` telemetry records carried no `span_id`/`parent_span_id`, so RPC round-trips
   could not chain into the span tree. Fixed in the inbox family: `InboxBase.InboxMetadata`
   gains `spanId`/`parentSpanId`; all four inboxes (`AsyncInbox`, `FutureInbox`,
   `AsyncMultiInbox`, `FutureMultiInbox`) populate them — span_id = the callee's span from the
   RPC reply, parent_span_id = the caller's span captured from the outbound request.
   Verified live against Eric's composable-example→lambda-example declarative demo (one
   continuous cross-app span tree) + new regression `PostOfficeTest.rpcTelemetryCarriesSpanLineage`.
   Note: Eric's staged `examples/lambda-example/app-log-context.yaml` rode into this commit
   (disclosed; he continued iterating on it, so no amend).

3. **Application log context ON by default (`9f9050e1`).** Eric, after a leadership call:
   the app-log-context feature should be enforced by default. Implementation follows the
   `default-rest.yaml` precedent — a **distinctly named** built-in
   `system/platform-core/src/main/resources/default-log-context.yaml` (cid/traceId/tracePath/
   spanId/parentSpanId/service/timestamp) with explicit code fallback in
   `LogContextConfig.loadConfigFile()`, rather than shipping a same-named
   `app-log-context.yaml` in the platform-core jar (same-name classpath shadowing across jars
   is classloader-order-dependent; the code fallback is deterministic everywhere).
   Application override = provide your own `app-log-context.yaml` (replaces the template
   entirely); opt-out = `app.log.context=false` (default `true`, read live from System

Scope note: enforcement manifests only in the structured JSON appenders
