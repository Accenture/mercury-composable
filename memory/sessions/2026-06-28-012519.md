# Session (2026-06-28T01:25:19.000Z)

**Agent:** Claude Code

Root-caused and fixed a telemetry/span-propagation bug in platform-core, found while validating the
sync-over-async worked example's trace (Eric noticed `sync.await` had no `parent_span_id`).

## Root cause
The per-function trace lives in `EventEmitter.traces`, a map keyed by **`Thread.currentThread().threadId()`**
(`startTracing`/`getTrace`/`stopTracing`). `WorkerHandler.processEvent` runs the function then `stopTracing`
removes the entry. A **Mono-returning** task returns *before* it completes, so its completion callback runs
later on the reactor **executor thread** — where `getTrace()` misses the (wrong-thread-keyed, already-removed)
entry. The response then carried no `span_id`, so `TaskExecutor` could not set the next task's
`parent_span_id` → orphaned span (e.g. `sync.await`). Cross-pod hops were fine because
`SimpleKafkaNotification` reads the span *inside* `handleEvent`, on the worker thread.

## Fix
`WorkerHandler.handleMonoResponse` now captures the `TraceInfo` on the worker thread before subscribing and
applies it (new `applyTraceContext` helper) when the Mono completes, instead of re-fetching `getTrace` on the
executor thread. The error path uses the same captured context. Regression test
`WorkerHandlerTest.monoResponseForwardsSpanId` (RED before, GREEN after). platform-core 350 tests pass; the
sync-over-async-demo fat jar rebuilt with the fix for manual re-test.
