### Evaluation
`ElasticQueue` (platform-core) is the reactive back-pressure overflow buffer behind every `ServiceQueue`
(20 in memory, then spill). BDB JE (`com.sleepycat:je:18.3.12`) is used *only* here and at a fraction of
its capability — non-transactional, non-durable (wiped on restart), FIFO-only, no random access: a
transactional B-tree serving as a transient disk FIFO. Lighter alternative: a **per-route append-only
segmented FIFO** (`[4-byte len][payload]`, advance read offset, delete segment on last-record-consumed).

### Drivers (Eric)
1. Performance — BDB's background checkpointer/log-cleaner occasionally block the Vert.x event loop.
2. Complexity reduction (simple = robust).
3. **Cross-language portability** — Mercury → Rust/others; BDB is JVM-only, a length-prefixed segmented
   FIFO is a language-agnostic format. (Primary long-term driver.)

### Design decisions (all resolved)
- Preserve the `ElasticQueue` API exactly — swap internals of one class (low blast radius); existing test is the gate.
- **Segmentation: YES** (bounds sustained-overflow disk; O(1) segment delete; no cleaner thread).
- **Transitional `elastic.queue.store = bdb | file` switch: YES** — pipeline-flippable fallback; both coexist P1→P3, BDB retired P4.
- `elastic.queue.cleanup` route + `deferred.commit.log` kept while the switch/BDB exists; retired with BDB at P4.
- **Q (Eric): run ElasticQueue I/O on virtual threads instead of the event loop?** Viable + VT-aligned, but a
  separate, larger change (touches `ServiceQueue` threading + a loop→VT hand-off). Captured as **optional P5**:
  keep the non-blocking fast path on the loop; move only the blocking spill to a per-route VT; gated on whether
  on-loop file I/O still shows in the tail after P3. Decoupled from the P1–P4 internals swap.

### P0 empirical baseline (`ElasticQueueBenchmark`, platform-core test src, uncommitted)
Sustains spill so every write/read hits BDB; measures inline op latency (= event-loop block time, since
`ServiceQueue` calls write/read inline on the loop).
- Fast box (1KB, 10k backlog, 130s): write p50 ~1µs, p99.9 0.11ms, **max 32ms**; ~10 stalls>20ms (occasional).
- Under pressure (4KB, 80k backlog, **512MB heap**, 180s): write p99.9 **1.6ms**, **read max 98ms**, 3,339
  ops>10ms, **53 stalls>20ms — continuous, clustering into housekeeping bursts** (e.g. 98/74/73ms back-to-back).
- Median is always ~1–3µs (the trap); the **tail** is the problem and scales with pressure. Throughput
  (72–160k ops/s) is ample — **not** the bar. Target for `file`: flatten p99.9/max toward p50.
