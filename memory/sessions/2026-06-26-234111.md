## Summary

Reworked the Kafka DLQ design after Eric pushed back on the global-DLQ knob added earlier today, and asked me
to **research industry best practice before fixing**. Web research (Confluent, Spring Kafka, Redpanda, Aiven,
KAFKA-3450) confirmed his instinct and reshaped the design:

- **Per-topic DLQ with a suffix is the industry standard** (Spring's `DeadLetterPublishingRecoverer` →
  `<topic>.DLT`; `.dlq`/`-dlq` equally common). Per-topic preserves each source's schema → reprocessing works.
- **Global/shared DLQ is an anti-pattern** (mixed schemas make replay hard) — exactly Eric's point.
- **Kafka auto-creation is off by default in prod**; producing to a missing topic blocks `max.block.ms`≈60s
  then fails. → DLQ topics must be pre-provisioned.
- **The real defect** was committing after a *failed* DLQ write (silent loss), not "where the DLQ points."
  Best practice: don't commit unless the message is durably stored.

### Changes (replaces the global `kafka.flow.dlq.topic` from the prior session)

- **Reverted** `kafka.flow.dlq.topic` (global). DLQ is strictly per-topic `<topic><suffix>`.
- **Configurable suffix** `kafka.flow.dlq.suffix` (default `.dlq`; e.g. `-DLQ`, `.DLT`). Blank → falls back to
  `.dlq` so the DLQ can never equal the source topic (self-loop guard).
- **Confirmed DLQ write + commit-gating.** New `KafkaRequestPublisher.publishSync` blocks on broker ack and
  throws on failure; `KafkaFlowConsumer.writeToDeadLetter` returns success, and `routeToFlow` commits **only**
  if the DLQ write succeeded — else does **not** commit (message redelivers, no loss) with a loud error
  pointing at pre-provisioning. The outbound notification path stays drop-n-forget (unchanged).
- `RetryPolicy` record field `deadLetterTopic` → `dlqSuffix`. Javadoc on consumer/autostart/record documents
  the per-topic rule, pre-provisioning requirement, and the anti-pattern rationale.
