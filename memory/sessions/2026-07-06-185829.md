## Field build failure — Guava `failureaccess` `NoClassDefFound` (fix on `main`, pending 4.6.2)

Eric reported the v4.6.1 field build failing (Snyk passed) with
`java.lang.NoClassDefFoundError: com/google/common/util/concurrent/internal/InternalFutureFailureAccess`
in `minimalist-kafka`'s `SchemaCodecCsfleConfigTest` (Confluent JSON serializer → Guava `LocalCache`).

**Root cause (Maven nearest-wins, not a 4.6.1 regression):** `InternalFutureFailureAccess` lives in
`com.google.guava:failureaccess`. In minimalist-kafka, Confluent's `kafka-avro-serializer` depends on
Guava **directly** → `guava:32.0.1` at **depth 2**, which beats platform-core's `guava:33.5.0` at
**depth 3** (`event-script-engine → platform-core → guava`). So the module ran Guava `32.0.1` and its
`failureaccess:1.0.1` arrived only as a transitive-of-a-transitive — the fragile node a strict/curated
field resolver drops. **Could not reproduce locally** (a plain `mvn` build resolves `failureaccess` fine;
the drop is field-resolution-specific).

**Fix (4 poms, `b47fac39` on `main`):**
- `minimalist-kafka`, `sync-over-async`: pin Guava `33.5.0` (dependencyManagement, overrides depth
  mediation) + declare `com.google.guava:failureaccess:1.0.3` **explicitly** (depth-1, immune to
  mediation/curation; also propagates downstream).
- `kafka-demo`, `sync-over-async-demo`: pin Guava `33.5.0` for version consistency (failureaccess already
  propagates from the modules).
- **Verified:** minimalist-kafka 79 + sync-over-async 32 tests green; demos build clean; reactor-wide
  sweep shows **no module on `32.0.1`** (all `33.5.0` + `failureaccess 1.0.3`).
